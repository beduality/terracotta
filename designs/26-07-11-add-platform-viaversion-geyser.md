# Proposal: Protocol Translator and Cross-Platform Compatibility Detection

**Date**: 2026-07-11  
**Status**: Draft  
**Related**: Provider-Specific Logic Layer, Override Pattern, Game Version Metadata Detector

## Summary

Extend Terracotta's metadata detectors to recognise when a project **explicitly integrates** with protocol translation or cross-platform compatibility layers (ViaVersion, Geyser, Floodgate, ProtocolLib, etc.).

These integrations are surfaced as **compatibility hints** (tags or metadata fields) rather than as primary loaders, because the underlying host loader (Paper, Velocity, etc.) remains the actual publishing target.

## Problem Statement

A developer writing a plugin or mod can target a compatibility layer in two ways:

1. **Implicit runtime dependency**: the server administrator installs ViaVersion/Geyser in the runtime `plugins/` folder. This is invisible to the project sources and cannot be detected.
2. **Explicit integration**: the developer declares an API dependency, adds a `softdepend`, or bundles a config for the compatibility layer. This *is* visible and useful metadata.

Today's detectors ignore the second category. A ViaVersion add-on is classified only as `paper` or `velocity`, and a Geyser/Floodgate plugin is classified only by its host platform. This loses information that helps users discover and trust cross-platform projects.

| Project Type | Signal We Can Actually See | What Terracotta Misses Today |
|--------------|---------------------------|------------------------------|
| ViaVersion add-on | `com.viaversion:viaversion-api` dependency | No hint that the plugin extends ViaVersion |
| Geyser add-on | `org.geysermc:geyser-api` dependency | No hint that the plugin targets Bedrock players |
| Floodgate plugin | `org.geysermc:floodgate-api` dependency | No hint of Floodgate / Bedrock integration |
| ProtocolLib-dependent plugin | `depend: [ProtocolLib]` in `plugin.yml` | Only the Bukkit/Paper loader is detected |

## Scope & Non-Goals

**In scope**

- Detecting explicit integration signals in project sources (build files, plugin descriptors, resource configs).
- Surfacing those signals as Terracotta metadata fields or tags.
- Keeping the host loader as the actual publishing target.

**Out of scope**

- Detecting what is installed in an end-user server's `plugins/` folder. That runtime state is not part of the project being published.
- Treating ViaVersion/Geyser as primary loaders. They are not platforms a plugin is *built for* in the same sense as Paper or Velocity; they are compatibility layers *on top of* those platforms.
- Modifying platform APIs to accept `viaversion` or `geyser` as loader slugs. Provider mapping will collapse these hints to the host loader.

## Goals

1. Detect explicit ViaVersion/Geyser/Floodgate/ProtocolLib integration from project sources.
2. Surface detected integrations as **metadata hints** (e.g. `tags` or a new `compatibility` field) without changing the canonical loaders list.
3. Extract supported Minecraft version ranges from the underlying host loader, not from the compatibility layer itself.
4. Keep detection opt-in and extensible via the existing `ProjectMetadataDetector` registry.

## Research Findings

### ViaVersion ecosystem

- **ViaVersion API**: `com.viaversion:viaversion-api` dependency. Used by plugins that hook into ViaVersion to send multi-version packets.
- **ViaBackwards / ViaRewind**: similar API artifacts under `com.viaversion`.
- A project that merely *runs alongside* ViaVersion (runtime dependency) does **not** declare this API and will not be detected.

### Geyser / Floodgate

- **Geyser API**: `org.geysermc:geyser-api`. Used by add-ons that extend Geyser behavior.
- **Floodgate API**: `org.geysermc:floodgate-api`. Used for Bedrock player linking, skin handling, etc.
- A `config.yml` under `Geyser/` is generally a *runtime* file generated by Geyser, not a project source file, so it should not be used as a primary signal.

### ProtocolLib

- Declared via `depend: [ProtocolLib]` or `softdepend: [ProtocolLib]` in `plugin.yml`.
- Strong signal for loader = `bukkit`/`paper` and environment = `server`, but does not change the loader itself.

## Proposed Changes

### 1. Compatibility Metadata Field

Introduce a narrow, optional field on `TerracottaProjectMetadata` for compatibility hints:

```kotlin
@Serializable
enum class TerracottaCompatibilityHint(val displayName: String) {
    @SerialName("viaversion")
    VIAVERSION("ViaVersion"),

    @SerialName("geyser")
    GEYSER("Geyser"),

    @SerialName("floodgate")
    FLOODGATE("Floodgate"),

    @SerialName("protocol_lib")
    PROTOCOL_LIB("ProtocolLib"),
}

@Serializable
data class TerracottaProjectMetadata(
    val loaders: List<String> = emptyList(),
    val gameVersions: List<String> = emptyList(),
    val environment: TerracottaEnvironment? = null,
    val compatibilityHints: List<TerracottaCompatibilityHint> = emptyList(),
)
```

`compatibilityHints` are **not** loaders. They do not change the host loader and are not sent directly to providers that do not understand them.

### 2. Compatibility Hint Detectors

Add a new detector that inspects build files and plugin descriptors for explicit integration signals:

```kotlin
class CompatibilityHintDetector : ProjectMetadataDetector {
    override fun detect(context: ProjectMetadataContext): TerracottaProjectMetadata? {
        val hints = mutableListOf<TerracottaCompatibilityHint>()

        if (hasDependency(context.cache, "com.viaversion:viaversion-api")) {
            hints += TerracottaCompatibilityHint.VIAVERSION
        }

        if (hasDependency(context.cache, "org.geysermc:geyser-api")) {
            hints += TerracottaCompatibilityHint.GEYSER
        }

        if (hasDependency(context.cache, "org.geysermc:floodgate-api")) {
            hints += TerracottaCompatibilityHint.FLOODGATE
        }

        if (dependsOnPlugin(context.cache, "ProtocolLib")) {
            hints += TerracottaCompatibilityHint.PROTOCOL_LIB
        }

        return if (hints.isEmpty()) null else TerracottaProjectMetadata(compatibilityHints = hints)
    }
}
```

### 3. Dependency-Aware Build File Source

Reuse / extend the existing build-file scanning with a small helper:

```kotlin
object BuildFileDependencySource {
    fun hasAny(cache: ProjectFileCache, coordinates: List<String>): Boolean {
        val text = cache.read("build.gradle.kts")
            ?: cache.read("build.gradle")
            ?: cache.read("pom.xml")
            ?: return false
        return coordinates.any { text.contains(it) }
    }

    fun hasAny(cache: ProjectFileCache, vararg coordinates: String): Boolean =
        hasAny(cache, coordinates.toList())
}
```

### 4. Plugin Descriptor Helper

```kotlin
object PluginDescriptorSource {
    fun dependsOn(cache: ProjectFileCache, pluginName: String): Boolean {
        val content = cache.read("src/main/resources/plugin.yml") ?: return false
        // Naive scan; YAML parsing can be introduced if the rest of the codebase already pulls a parser.
        val pattern = Regex("""(?:depend|softdepend|loadbefore|loadafter)\s*:\s*\[[^\]]*\\b$pluginName\\b[^\]]*\]""")
        return pattern.containsMatchIn(content)
    }
}
```

### 5. Provider Mapping

Compatibility hints do not alter the canonical loader list. During provider transformation they can be:

- **Dropped** for providers that do not support tags (e.g. CurseForge).
- **Mapped to tags/categories** for providers that support free-form tags (e.g. Modrinth).

| Detected Hints | Host Loader | Published Loader | Optional Tag |
|----------------|-------------|------------------|--------------|
| `viaversion` | `paper` | `paper` | `viaversion` |
| `geyser` | `paper` | `paper` | `geyser` |
| `floodgate` | `velocity` | `velocity` | `floodgate` |
| `protocol_lib` | `paper` | `paper` | `protocol-lib` |

## Implementation Plan

1. Add `TerracottaCompatibilityHint` enum and extend `TerracottaProjectMetadata` with `compatibilityHints`.
2. Add `BuildFileDependencySource` helper.
3. Add `PluginDescriptorSource` helper.
4. Add `CompatibilityHintDetector` and register it alongside existing detectors.
5. Add unit tests covering:
   - Detection from Gradle/Maven dependencies.
   - Detection from `plugin.yml` depend/softdepend.
   - No detection when only runtime compatibility is implied (no source signal).
6. Update provider-specific logic layer to map hints to platform tags where supported.
7. Document supported compatibility hints in configuration reference.

## Risks & Considerations

1. **False positives from `config.yml`**: a generic `config.yml` resource is not enough to infer Geyser integration.
   - **Mitigation**: Only detect Geyser from explicit `org.geysermc:geyser-api` / `floodgate-api` coordinates, not from resource files.

2. **Runtime-only ViaVersion usage**: many plugins simply work alongside ViaVersion without using its API.
   - **Mitigation**: Do not detect runtime-only usage. The proposal only detects explicit API or dependency declarations.

3. **Provider incompatibility**: some platforms have no tag system.
   - **Mitigation**: Hints are optional metadata and are dropped when unsupported.

4. **Coordinate drift**: Maven coordinates may change.
   - **Mitigation**: Keep coordinate lists in small constants and add a sample-project integration test.

## Benefits

1. **Accurate metadata** for cross-platform and compatibility-layer projects.
2. **No loader pollution**: Paper/Velocity/etc. remain the canonical loaders.
3. **Backward compatible**: existing detection is unchanged when no hints are present.
4. **Extensible**: future compatibility layers (PacketEvents, BKCommonLib, etc.) can add new enum values and detectors.
